h1. Quay Operator: Clair Startup Race Condition Causing Pod Restarts

h2. Summary

Clair app pods fail to start on first attempt and require 1-4 restarts due to a race condition where the Clair application attempts to connect to ClairPostgres database before the database is ready to accept connections.

*Severity:* Medium
*Impact:* Increases Quay deployment time by 30s-2min, wastes resources through unnecessary pod restarts
*Reproducibility:* 100% on fresh Quay deployments
*Component:* quay-operator v3.15.2

----

h2. Problem Description

When deploying a QuayRegistry with Clair security scanning enabled, the Clair application pods consistently fail on first startup with database connection errors, requiring Kubernetes to restart them 1-4 times before successful initialization.

h3. Root Cause

*Race Condition:* The Quay Operator creates ClairPostgres and Clair App Deployments without ensuring the PostgreSQL database is ready to accept connections before Clair attempts to connect.

*Missing Dependency Management:*
* (x) No initContainer to wait for postgres readiness
* (x) No startup dependency enforcement by Quay Operator
* (x) Clair app fails fast instead of retrying connection

----

h2. Why Changing QuayRegistry Component Order Won't Fix This

Current QuayRegistry spec shows:
{code:yaml}
spec:
  components:
  - kind: clair          # app listed first
    managed: true
  - kind: clairpostgres  # postgres listed second
    managed: true
{code}

{panel:title=Key Finding|borderStyle=solid|borderColor=#ccc|titleBGColor=#FFFFCE|bgColor=#FFFFCE}
*Even when Operator creates postgres Deployment BEFORE app Deployment (as seen in Cluster 1), the race condition still occurs* because:
* Operator doesn't wait for postgres to be Ready
* Operator doesn't check if postgres is accepting connections
* Pod startup timing depends on scheduler, not YAML order
{panel}

h2. Proposed Solutions

h3. Solution 1: Add initContainer to Clair App (Recommended - Quick Fix)

{code:yaml}
initContainers:
- name: wait-for-postgres
  image: registry.redhat.io/rhel8/postgresql-15:latest
  command:
  - /bin/bash
  - -c
  - |
    until pg_isready -h registry-clair-postgres -p 5432 -U postgres; do
      echo "Waiting for postgres..."
      sleep 2
    done
{code}

*Benefits:*
* (/) Can be implemented in Quay Operator quickly
* (/) Uses PostgreSQL's own readiness check
* (/) Low risk, high reliability

h3. Solution 2: Enhance Quay Operator (Long Term - Best Solution)

Modify Operator reconciliation logic to:
# Create ClairPostgres Deployment
# Wait for ClairPostgres Deployment to be Available
# Verify postgres Service has endpoints
# Only then create Clair App Deployment

*Benefits:*
* (/) Eliminates race condition at source
* (/) Proper dependency management
* (/) Can be extended to other components

h3. Solution 3: Improve Clair Connection Retry (Upstream)

Add exponential backoff retry logic to Clair application when connecting to database.

*Benefits:*
* (/) Makes Clair more resilient
* (/) Benefits all Clair deployments
* (/) Cloud-native best practice

h2. Steps to Reproduce

{code:bash}
# 1. Deploy fresh QuayRegistry with Clair
oc apply -f - <<EOF
apiVersion: quay.redhat.com/v1
kind: QuayRegistry
metadata:
  name: registry
  namespace: local-quay
spec:
  components:
  - kind: clair
    managed: true
  - kind: clairpostgres
    managed: true

# 2. Watch Clair app pods
oc get pods -n local-quay -l quay-component=clair-app -w

# 3. Check restart count
oc get pod -n local-quay <clair-app-pod> -o jsonpath='{.status.containerStatuses[0].restartCount}'

# Expected: restart count = 1-4
{code}
